在我从事技术领域的时间里，我学到了你懂什么工具或语言真的不重要。在你在这个领域任职期间，你将学会和放弃多种工具/语言。在你的旅途中，你会记得的是:

*   你一路走来学到的教训
*   你的观点
*   你的方法

这三点是让有经验的工程师如此受欢迎的原因。我们都听说过这样一句话，“如果你只有一把锤子，所有的东西看起来都像钉子。”同样的逻辑也适用于做工程师。

你可以毫无问题地掌握一门新的语言或工具，但是知道适合这项工作的合适工具要困难得多。

考虑到这一点，我想分享我在 10 多年里学到的一些重要的东西，这些东西可以帮助你作为一名工程师加速成长。

# 建立模式

“建立模式”是什么意思？这意味着不要只解决今天的问题，也要解决明天的问题。我在网上简单搜索了一下，维基百科是这样描述的:

> 在[软件工程](https://en.wikipedia.org/wiki/Software_engineering)中，**软件设计模式**是一种通用的、[可重用的](https://en.wikipedia.org/wiki/Reusability)解决方案，用于解决给定环境中常见的问题

在我们进一步解释之前，让我们举一个真实的例子:

“我在笔记本电脑/工作站上运行代码时遇到了问题，因为 X 没有安装或者版本不正确。”

我可以想象我们都曾在某个时候遇到过这种情况。也许:

*   一位同事正在使用不同版本的库/二进制文件
*   添加了一个新的依赖项，但您忘记了安装它
*   你是一名新员工，正试图弄清楚如何让一个应用程序工作。

“常识”的解决方案是安装或升级任何引发错误的东西，对吗？

那种方法只能解决眼前的问题。也许，您的笔记本电脑上缺少其他依赖项。即使今天这个问题被修复了，也不能保证下次有人更新代码库时它不会发生。每次你雇佣一个新的工程师，他们也必须经历同样的痛苦。

相反，你能做些什么来永久解决这个问题呢？不仅能解决眼前的问题，还能防止它发生在你或其他人身上。

你可以构建一个 runner 或者 docker 镜像，它会自动安装你所有的依赖/工具/脚本等等。每次你运行这个容器，你都会得到一致的体验。如果您的同事添加了一个新的依赖项，它会自动安装。如果你雇佣了一个新的工程师，他们可以下载容器并立即开始增值。我们甚至没有提到代码可以基于您的操作系统以不同方式运行的事实。有了容器，每个人都使用相同的操作系统。如果这个容器是一个应用程序，这也意味着每个人都在同一个映像上开发，最终将在生产中运行！

一旦给出答案，这个例子看起来非常简单明了，但是今天许多人仍然在他们的本地机器上运行`npm install`或`brew install X`。

*建立模式是最难也是最容易做到的事情，而且影响最大。*模式很难，因为很难定义一种你从未采用过的方法。如果您从未听说过 docker 或 runner，您如何知道可以使用它来解决这个问题呢？这就是经验的可贵之处。但是，建立模式也是最容易做到的，因为互联网就在你的指尖。我坚信，如果我试图解决一个问题，别人已经克服了这个障碍。也就是说，你仍然需要评估你找到的解决方案是否适合你的情况。如果解决方案不适合你，你仍然可以从中获得灵感，或者至少知道已经尝试了什么。

这也是我强烈建议经常阅读文章、与工程师同事交谈或参加会议的地方。你可能会想出如何改进你甚至不知道可以改进的东西。

值得一提的是，与 5 分钟修复相比，构建跑步者需要几个小时，但是:

*   想想你因为不再有这个问题而节省的时间
*   想想你为同事节省的时间
*   想想为您和整个团队添加新工具变得多么容易
*   考虑到容器也可以作为您的 CI/CD 管道的一部分来运行测试或做任何您想做的事情。
*   最后，记住上下文切换是昂贵的。每次都是 5 分钟的修复，然后在你被打断之前，再花 30 分钟回到正确的心态。

我曾经认识一位杰出的工程师，与他的同行相比，他总是能写出两倍多的代码。代码运行良好，质量很高，几年后，我知道它仍在某处使用。他告诉我，他只花了 20%的时间编写代码。他其余 80%的时间都在思考/处理他想做什么以及为什么要做。让它沉下去吧。

# 不要害怕为正确的工作使用正确的工具。

这直接回到了我们在本文开头提到的锤子/钉子类比。不管你用锤子有多熟练，它永远不会是扳手。

如果我在这里提供一个例子，它可能太具体而不能提供价值。为了让您有个概念，在我们的自动化中，我们利用了:

*   舵
*   舵手
*   雄蜂
*   将（行星）地球化（以适合人类居住）
*   python 脚本
*   戈朗
*   自定义 docker 图像
*   等等…

不同的工具在不同的事情上发光，用起来没问题。现在我确实想澄清我们在做什么和不在做什么。

## 我们是:

*   使用给定任务的最佳工具
*   在相同的工作中始终使用相同的工具
*   使用辅助工具(使事情完全自动化)
*   使用带有声明性语言的工具，其中配置可以在代码中定义(这是版本化的，并经过同行评审)
*   使用幂等的工具/脚本

## 我们不是:

这整个部分只是为了突出“**始终如一地使用相同的工具完成相同的工作**”。可以使用互补工具，但不能使用竞争工具(例如云的形成与地形)。

一旦我们决定一个工具将被用于 X，那就是我们用于 X 的唯一的**工具**

如果我们决定使用 Terraform 来部署我们的基础架构，它将是我们用于基础架构的唯一工具，不需要云的形成，不需要手动部署，什么都不需要！！！一旦我们决定使用 helm 部署 Kubernetes 包，你不会发现我们使用原始 YAML 部署任何东西。

记得我们讨论的第一件事，“建立模式”。如果您每次都使用不同的方法来解决一个重复出现的问题，这会给自动化带来不必要的摩擦，并且通常会令人困惑。它还会导致重复工作，并且意味着无论何时做任何事情，都必须考虑多种方法，而不是一种方法。

## 附录:

定制工具和脚本——完全可以在内部构建自己的工具。不过，在做出决定之前，你应该先检查一下是否有人或某家公司已经这么做了。定制工具和脚本总是以非常浪漫的方式被提及。“哦，我可以让它做我想做的任何事情，无论我想做什么，无论我何时想做什么”。虽然这种说法是非常正确的，但是从头构建任何东西也是非常复杂的，并且引入了使用由社区或公司维护的工具时所没有的开销。对于您创建/维护每一个定制工具，这个问题只会变得更加严重。

同样，定制工具是好的，只要确保它是用于正确工作的正确工具(并且值得花费/开销)。

# 自动化一切

我听说亚马逊的工程师们使用了一个术语，这个术语多年来一直困扰着我，“无差别重物”。为了保持增长，亚马逊不能只是不断增加员工。相反，他们不断地识别必要的、重复的、但是在迭代之间不会改变的任务，并找到一种方法来自动化它。你可以轻松地做同样的事情。

一个真实的例子是:“每次我建立一个 AWS 帐户，我都必须查找我需要的 AMI IDs 来配置我的基础设施。”

这项任务相当简单明了，但这是一个手动步骤:

*   有人必须记住这一步是必要的
*   实际上必须有人去做
*   除了“解除”自动化之外，它没有提供任何价值

我工作的一个工程师遇到了这个问题，他说“我不想再做这个了”。他们在 Terraform 中创建了一个数据资源，可以自动查找 ami。从此以后，我们不必再去查 AMIs，相反，我们可以专注于更大的问题。

# 保持增长

如果你在科技领域呆过一段时间，你就会知道它在不断变化。极限不断被打破。语言来来去去或者被重新发明。几年前，Kubernetes & Docker 还不是一个真正的东西，但现在它们无处不在。随着它们的采用，也出现了大量的工具和应用程序。像技术一样，你也必须不断发展。这意味着几件事:

*   永远不要停止阅读文章、参加会议或学习
*   不要带着“我是高级工程师”或“我是对的”的想法去争论。实际上听听这个论点，如果你的方法不是最好的，那就从中学习。即使你最终是“对的”，这仍然是一个帮助别人成长的机会。
*   定期查看旧项目，看看你在哪些地方可以做得更好。
*   设置新工具和语言的概念验证，并进行测试。即使事实证明你当前的语言/工具更好，你仍然能够清楚地说明为什么它们更好，而不是说“这是我们一直使用的”。
*   不要因为“我们一直是这样做的”而害怕改变。仍然考虑投资回报率，但如果改变有意义，就去做。

这个特定的主题是很好理解的，我相信许多文章都是就这个主题写的，所以我就不赘述了。继续成长吧！

# 结论

到目前为止，您可能已经注意到，所有这些部分都以某种方式相互联系在一起。这是有原因的。我希望传达的是，成为高级工程师的关键之一是你的心态。建立模式、使用正确的工具和自动化是齐头并进的。没有知识，你也不能做这些事情。

每一天，你都会有无数的分心或小问题试图把你的注意力从你试图解决的大问题上转移开。有时分心感觉像被一千个剪纸杀死。这些“剪纸”会影响你的工作效率；他们也会影响整个团队的生产力。

下次你遇到问题时，请记住这些要点。它们会帮助你做出更好的决定，通过足够的重复，也会帮助你成长。
