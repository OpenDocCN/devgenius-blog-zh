我们想写人们想用的软件。因此，对它们的要求必须是完整的。

在这篇文章中，我们将看看我们应该问的一些问题，看看这些需求是否有意义。没有完整的需求，我们无法进行。

# 指定的竞争属性之间的权衡是否可接受？

为了完成项目，我们必须做出的权衡必须是用户愿意接受的。

这是时候站在用户的立场上，问他们是否真的喜欢这些权衡。

没有用户认为可以接受的权衡，很难做出最终可以接受的东西。

# 需求是否避免指定设计？

我们不想要的是把需求文档变成技术设计文档。

因此，我们必须避免项目中的任何技术细节。

我们必须在单独的文件中详细说明设计。

需求文档应该从用户的角度详细说明一切。

# 需求有一致的详细程度吗？

我们应该询问需求是否足够详细，以便我们可以继续实现。

如果有些事情不清楚，那么我们必须在继续之前澄清它们。

如果某件事太详细，我们可能会要求简化，只要它仍然清楚。

# 需求是否足够清晰，可以交给一个独立的小组来执行？

如果我们想要开始实现，那么我们必须确保我们从需求文档中读到的信息足以让我们继续前进。

如果没有足够的细节，那么我们必须修改文档，以包括更多的细节和澄清事情。

否则，我们可以着手实施。

# 每个项目都与问题及其解决方案相关吗？

我们需要知道列出的每个需求是否真的与解决用户的问题相关。

这意味着我们必须考虑是否所有的事情都足够相关，这样我们才能继续实现列出的所有需求。

如果存在与解决手头问题无关的额外需求，那么我们必须删除它们。

# 每个需求是否可测试？

需求应该是可测试的。如果不是，那么我们将很难向用户展示我们的产品如何解决他们的问题。

每一个项目都有自己的输入和结果输出，我们可以检查。

# 是否规定了所有可能的需求变更？

如果我们现在知道了需求的可能变化，那么应该详细说明这些变化。

否则，如果它改变了，那么我们应该立即更新每个人，这样开发人员就不会做对满足需求没有帮助的无用工作。

我们还应该详细说明变化的可能性，以便每个人在变化来临时都有所准备。

# 在文档开始开发之前，信息是不可用的。

如果需求文档中的某些部分对于实现来说不够清晰，那么我们每个人都必须知道，这样开发时间就可以花在准备开发的东西上。

这是另一种没人花时间做会被扔出去的事情的方式。

# 如果我们的实现满足每一个需求，它会被接受吗？

这是我们在实施之前应该知道答案的一个问题。

如果用户在我们完成工作后不会高兴，那么我们可能不应该做。

这是因为如果我们在做不知道用户是否喜欢的东西，我们只是在浪费时间做无用的事情。

# 我们对所有要求都满意吗？

如果不是每个人都对需求感到满意，那么在我们开始实现之前，我们必须确保每个人都满意。

因为如果有人对他们感到不舒服，那么我们可能会有我们可能还没有想到的问题。

在我们做任何事情之前，最好把事情想清楚。

# 结论

如果我们不想放弃我们为实现所做的任何工作，那么在开始之前，我们必须考虑我们正在实现的特性的需求。

否则，我们以后可能会遇到问题，浪费时间和精力做我们不需要的事情，或者因为我们没有充分考虑需求而被扔掉。
