# 通过紧密的反馈循环改进代码重构

如果你面临重新分解你或其他人写的一些代码，这可能是一项艰巨的任务。说了这么多，是极其普遍的；事实上，有一本非常受欢迎和著名的书叫做*重构:改进现有代码的设计，作者是马丁·福勒*，该书通过系统地列举如何实现它的策略而深入探讨了这一点。

在这篇文章中，我们将回顾福勒在那本书中提到的一点，那就是当你重构代码时要有紧密的反馈循环。

这里的想法是，当你重新分解时，你会想要做一些小的改变。根据 Fowler 的说法，在每一个小的改变之后，你会想要*立即编译和测试*以确保你没有破坏任何东西。“错误很容易犯——至少我发现它们很容易犯，”他说。

这里的想法很简单:通过一个小的改变，你有一个更小的可能出错的事情的爆炸半径。你不必挖掘大量的变化来查看*到底是什么**破坏了功能。发现变化会更容易，解决变化也会更容易。您节省的调试时间将是非常重要的。*

*这是一个紧密反馈循环的关键:你能立即得到结果的小变化。从心理学的角度来看，知道你改变的东西起作用的内心平静可以让你放松，甚至在这个过程中感觉更好。*

*在这个反馈循环中，编译是很重要的，而功能测试在确认重构成功的过程中肯定可以发挥作用。福勒还指出，一套可靠的测试确实会收紧反馈回路。*

*福勒说，测试应该是自我检查，基本上报告所有绿色，与人工检查相反，节省了重要的击键和点击。*

*一旦您的测试通过，下一步就是将工作变更提交到您的本地版本控制系统，比如 git。这样，您将始终有一个提交的工作状态，以防以后出错；你只需要简单地回滚到那个时间点。Fowler 确保强调本地部分，因为这些是私有提交。然后，他亲自将他的更改压缩成更重要的提交，然后推送到共享存储库。*

*有了更紧密的反馈循环，重构将成为一个你更有信心的过程，有时这能带来很大的不同。*
