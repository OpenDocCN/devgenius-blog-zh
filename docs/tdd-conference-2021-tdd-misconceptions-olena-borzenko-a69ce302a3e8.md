# 我的个人笔记

# 第一个项目

*   TDD 既复杂又耗时。
*   她的一个 TDD 项目时间表非常紧，团队需要加快进度。
*   用户故事在不断变化。
*   他们不知道如何正确地编写测试。
*   TDD 的目的没有被很好地传达。
*   “发现了窃听器？修好它。写一个测试来证明它是固定的”方法。
*   他们的目标是 100%的代码覆盖率
*   大多数测试用例是无用的，目的在于覆盖率，而不是质量。
*   他们大量使用[模仿](https://dev.to/mcsee/code-smell-30-mocking-business-3glm)。
*   他们没有避免严重的缺陷。
*   完全模拟组合的无用案例只是为了达到 100%的覆盖率

# 第二个项目

*   这个更基于 BDD/行为。
*   只需编写通过测试所需的代码。
*   BDD 比 TDD 更容易实现。
*   BDD 之后，TDD 更容易进行。
*   TDD 帮助更好的设计。
*   TDD 将代码分成更小的片段。
*   TDD 使得代码更容易重构。
*   他们用了[黄瓜](https://cucumber.io/)。
*   通过使用 TDD，微软和 IBM 的预发布缺陷密度减少了 40%到 90%。
*   大多数时候，我们不需要重构阶段。
*   即使简单的代码也需要测试，因为测试也是规范。​
