理解软件架构(一个好的架构)

原文：https://blog.devgenius.io/understanding-software-architecture-and-a-good-one-eaca33090169?source=collection_archive---------0-----------------------#2019-10-10

软件架构指的是软件(系统/项目)的基本结构。

它基本上是一个计划，包含了要决定的软件系统的所有重要方面，这些方面极难改变或重新思考。

软件架构中的关键角色包括:

1.系统设计组织即软件系统将如何工作，它将遵循什么工作流程，软件系统中的模块将如何协同工作？等等。

2.系统部件通信即系统模块如何通信。

3.依赖关系即软件系统中的依赖关系是什么？如何实现依赖关系(具体的指导方针或技术)。

软件架构？为什么要跟着？

这个问题最简单的答案是“让实现/开发/编码阶段变得轻而易举(至少容易管理)”。

一个好的体系结构可以确保在实际实现系统时有最少的停顿。它有助于在未来轻松管理任何类型的更改或功能添加，以及系统的可伸缩性和可靠性。

确定好的和有效的软件架构的主要原因。 高效的沟通方式 :一个好的、容易掌握的软件架构提供了客户、管理层、开发人员等之间高效的沟通方式。同时，这也让每个人更容易理解这个系统。

2. 使早期规划和决策变得容易 : 在软件架构规划期间，做出了大量关于软件系统的决策，这些决策确保了系统的稳定性、可伸缩性、安全性等方面。这些决策对项目非常重要，并且在项目的后期阶段很难更改(很大程度上由开发人员更改)。(如果你从事过任何企业项目，你就会明白我这里的意思。)

正在总结(暂时:)

从开发人员的角度来看，一个好的架构可以确保项目易于开发和维护。

  1. 软件架构应该是健壮的，以便于更好地修复错误。
  2. 所有组件的工作应该特别让所有参与项目的人容易理解。
  3. 应能长期使用，即应能适应未来的要求和技术变化。
  4. 最重要的方面是可伸缩性。设计应该能够处理可伸缩性。
  5. 如果对系统进行任何更改，性能和速度不应受到影响。
  6. 系统应该是安全的。系统的安全级别应该在任何变化和使用中保持完美。
  7. 从用户的角度来看，该系统应该是用户友好的和高度可靠的。毕竟，在一个系统拥有的所有特性中，如果它不是用户友好的或可靠的，那么它就不可能获得用户的信任。
