> 可能你正在做一个过去的项目，或者很有可能你被要求去做它…

所以现在你需要浏览源代码，对项目的编码方式有一个基本的了解(这很糟糕，也很耗时)。

因此，问题不在于如何修复，而在于何时修复。尤其是如果你在中小型公司工作，开发人员实际上没有多少时间可以分配给文档(即使他/她准备了一份文档，也可能是一份糟糕的文档)。

**想象有一个截止日期，你需要让它生效，你直接开始记录错误，做一些调试，等等……**

![](img/c2acd2fc0a74eadc0bace6bc2b4e749e.png)

照片由[卢卡·布拉沃](https://unsplash.com/@lucabravo?utm_source=medium&utm_medium=referral)在 [Unsplash](https://unsplash.com?utm_source=medium&utm_medium=referral) 上拍摄

但是，在这种时刻，我们都会面临一些情况——(从容易到难)

1.  一些第三方代码最近更新，因此不支持某个特定的功能，但你猜怎么着，已经有一个他们通常提供的替代品
    > >我们只需要交换它
2.  [FRONTEND]看起来由于某种原因代码不起作用，你无法从逻辑上思考它。可能从 API 获取的数据并不像它应该的那样
    > >要么改变 I/P 数据，要么
    > >只是将错误转移到后端的数据
    (如果是全栈，那就太不幸了)
3.  过度设计的代码，我的最爱。
    > >把代码分成小而干净的片段，这样更容易一目了然，现在你可能知道解决方案了。
4.  最全面的抽象，因此你现在唯一知道的 2 件事是这是产生错误的东西，而且不知何故它是重要的(LOLZ)
    > >只关注输入和输出理解那个函数试图对输入做什么，以及它缺少什么，
    > >还有最好看一下下一个和上一个接触点(周围的组件)大概全能的上帝已经添加了一些注释
    **(反正这是折磨人的)【关键词】**
5.  现在，最糟糕的是，你发现了代码中的错误，但似乎没有什么工作，因为代码是好的。取而代之的是，由于别的原因，事情变得一团糟。
    (接受吧，要么是早餐不好/胃不舒服/等等，这一天不适合你)
    > >只是让项目负责人知道这需要时间…
    > >现在检查整个代码流，是什么搞乱了它
    (顺便说一句，如果通常最终会重写整个流程)

> 这些通常是我时常面临的问题。
