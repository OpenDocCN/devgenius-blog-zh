个人简历

Olena是德国柏林Adecco集团的全栈开发人员。她之前在乌克兰的一家服务公司工作，参与了从小型创业公司、B2B应用程序到企业平台的各种产品的创建。此外，她对新技术、干净的代码和最佳实践充满热情。在她的空闲时间里，当她不把时间花在爱好上时，她喜欢围绕现实生活中的用例建立演示，与他人分享知识，反之，了解别人的经验。

TL；DR: TDD需要承诺和沟通。人们必须认同它才能拥抱它。

讨论

我的个人笔记

第一个项目

  • TDD既复杂又耗时。
  • 她的一个TDD项目时间表非常紧，团队需要加快进度。
  • 用户故事在不断变化。
  • 他们不知道如何正确地编写测试。
  • TDD的目的没有被很好地传达。
  • “发现了窃听器？修好它。写一个测试来证明它是固定的”方法。
  • 他们的目标是100%的代码覆盖率
  • 大多数测试用例是无用的，目的在于覆盖率，而不是质量。
  • 他们大量使用模仿。
  • 他们没有避免严重的缺陷。
  • 完全模拟组合的无用案例只是为了达到100%的覆盖率

第二个项目

  • 这个更基于BDD/行为。
  • 只需编写通过测试所需的代码。
  • BDD比TDD更容易实现。
  • BDD之后，TDD更容易进行。
  • TDD帮助更好的设计。
  • TDD将代码分成更小的片段。
  • TDD使得代码更容易重构。
  • 他们用了黄瓜。
  • 通过使用TDD，微软和IBM的预发布缺陷密度减少了40%到90%。
  • 大多数时候，我们不需要重构阶段。
  • 即使简单的代码也需要测试，因为测试也是规范。​
