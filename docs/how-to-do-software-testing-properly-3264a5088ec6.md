写软件很难。如果我们没有测试来捕捉它们，事情很容易就坏了。

在这篇文章中，我们将看看如何测试我们的软件，在早期发现缺陷，以便我们可以快速修复它们。

# 如何测试

我们应该看看如何正确地测试，这样我们就不会错过任何东西。

测试需要做一些事情，包括回归测试、设置测试数据、使用 GUI 系统、测试测试和彻底测试。

我们还可以测试 ode 本身的设计以及我们用来构建软件的方法。

我们可以用一些指标来衡量我们的方法是否好。例如，我们可以检查我们写的代码行数，

度量可以给我们的代码一个及格分数。

我们必须找到合适的度量标准来衡量我们方法的成功，比如交付时间、缺陷数量等。

# 回归测试

回归测试在我们对系统做了一些事情之后检查输出。

我们必须运行它们来确保我们没有因为改变颂歌而破坏任何东西。

这些可以在我们修改代码时自动运行。

# 测试数据

我们可以使用真实世界的数据或合成数据来运行我们的测试。

我们两者都需要，因为它们的性质不同。真实世界的数据来自真实世界的来源。

它可能来自某个原型或竞争对手的系统，

它们为我们提供了不同的方式来揭示缺陷。

合成数据是人工生成的。我们需要生成大量数据来运行我们的测试。

此外，我们需要测试边界条件。

我们还需要展示一些统计特性的数据，如导致失败案例和边缘案例的数据。

# 使用图形用户界面系统

测试 GUI 需要专门的测试工具。我们可以使用 Selenium 或 Cypress 之类的东西来测试我们的 GUI，这样我们就可以像用户一样使用我们的应用程序来检查结果。

我们可以自动化大多数重复性的案例，但有些事情很难自动化，如上传和下载文件。

我们的系统还应该具有足够的解耦性，这样我们就可以在没有 GUI 的情况下测试应用程序逻辑。

一旦应用程序的逻辑得到验证，我们就可以隔离 GUI 上应该局限于 GUI 本身的任何错误。

# 测试测试

我们必须测试这些测试，因为它们也是我们生产的。

我们必须确定他们确实在捕捉回归和错误。

我们可能想任命一个项目破坏者来破坏我们的应用程序。他们可以复制代码，故意引入错误，然后看看我们的测试是否能发现它们。

# 彻底测试

我们必须确保彻底测试我们的产品。有测试覆盖分析工具可以做到这一点。

他们跟踪哪些代码行已经运行，哪些没有运行。

它让我们知道我们的测试有多全面，但是我们不应该期望 100%的覆盖率。

我们要识别程序所有可能的状态。这是我们知道我们实际上彻底测试了我们的代码的方式。

仅仅因为我们用测试运行了所有的代码行并不意味着我们彻底地测试了它。

# 测试状态覆盖率，而不是代码覆盖率

我们应该着眼于全面的状态覆盖，而不是着眼于代码覆盖。

这样，我们将确保除了通常的情况之外，边界和边缘的情况也包括在内。

# 何时测试

测试应该比截止日期前更早开始。

我们应该从头开始编写测试，这样我们每次修改代码时都可以运行它们。

测试应该会自动运行。结果应该被自动解释。

后台运行回归测试应该不成问题。

测试应该在预定的基础上运行。

# 收紧网

如果一个 bug 漏过一次，我们修复了它，那么我们应该为它写一个测试，这样同样的 bug 就不会再发生了。

一旦人类测试人员发现一个 bug，这应该是人类测试人员最后一次发现相同的 bug。

# 结论

我们应该确保我们的测试覆盖了所有的州。

测试应该自动运行，任何发现的 bug 都应该添加一个测试，这样就可以自动发现。

GUI 也需要用专门的工具来测试。
