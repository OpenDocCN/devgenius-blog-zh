# 无尽的测试

2021 年，我写了两篇文章，Java 中的单元测试(T0)和 T2 的权衡:质量(T20)速度(T3)来强调测试技巧的重要性。

这些天我致力于为我的副业项目的一些核心模块编写测试用例，以验证事务逻辑是否可以在不同的场景下照常工作。在这个过程中，我开始思考一个工程师面临着什么样的压力，让他远离测试工作，这似乎是必要的。然而，工程师们宁愿不去做，或者把它留给 QA。事实上，从时间表、业务规模或其他角度来看，有许多原因会影响工程师从事这项工作的愿望。

# 从表象深挖

在我看来，有三个主要原因使得测试任务难以推进。

*   **用例设计随着业务逻辑的复杂化而增长**

以一段代码为例，当我想在有限大小的列表中添加一个新的交易时，必须考虑两种情况。

**【1】**列表大小足够新交易消费
**【2】**列表大小不足，需要弹出前面一个；在其尾部添加一个新的交易

上面的例子是一个简单的例子，我们只需要验证它在这两种情况下是否有效。尽管如此，现实世界总是以一种意想不到的方式让人们失望。如果这个函数有更多的分支呢？你会增加更多的案例来验证它的正确性吗？

老实说，大多数人都愿意通过编辑足够的测试用例来确保他们的功能能够照常工作，这导致**测试时间比开发时间**长，因为他们需要覆盖所有可能的执行路径。

因此，用例的复杂性直接决定了在接下来的开发阶段是否会导致更长的测试工作。

*   **测试类别/视角随着业务逻辑的扩展而变化**

测试不仅关注功能的正确性，还关注它要测量的对象。例如，在交易系统中，延迟是一个大问题，而在购物网站中，吞吐量是主要问题。在真实的场景中，不仅仅有两个主题需要解决。这取决于客户关心什么。

如果只是尝试对测试种类进行分类，你会提到如下:
**【1】单元测试
【2】集成测试
【3】回归测试**

如果你想验证某些主题，你可能会想到以下几种:
**【1】性能测试(延迟、吞吐量、代码覆盖率等)
【2】行为测试**

对于一个高级工程师来说，他可能能够根据团队希望在不久的将来实现的业务目标来设计各种测试用例。此外，他可以很好地定义测试边界，防止不必要的域被验证所覆盖。

然而，当涉及到市场适应时，设计有用的测试场景需要更长的时间，所以成员应该在用例设计和发布计划之间达成妥协。

*   **时间线迫使工程师精心制作可工作的——而不是可测试的软件**

我想这可能是大多数人抵制测试工作的主要原因。截止日期总是存在的——所以他们宁愿尽快完成任务，而不是进一步测试是否完成了任务。

在台湾，软件开发很容易受到商业决策的影响，尤其是上市时间，所以你总是会得到可工作的产品，这给最终用户带来不友好的体验，因为它们不是完全可测试的。

> 我们应该首先将产品投放市场；测试用例可以在以后添加。

嗯，你可能会在别的地方听到类似的评论。通常在产品发布后，你就没有机会这样做了..因为您需要修复大量的错误，如果在最开始添加测试用例，这些错误是可以避免的。你把时间浪费在修补缺陷上，而不是提炼根本原因。

# 测试覆盖和真实业务之间的平衡

> 软件工程总是一种权衡，测试工作也是如此。

我完全支持任何业务都不应该为了满足市场需求而延迟，但这并不意味着我们应该牺牲基本的，甚至是重要的测试任务。应该达到平衡以实现对软件产品以及用户体验的基本保护。也许有些建议我们可以考虑。

*   **任务估计中的持续缓冲时间**

可工作的软件遭受来自用户的真实测试，因为它由于不充分的验证而不可测试。因此，强烈建议团队在所有的任务中维持缓冲时间，以便测试用例参与到开发周期中。

*   **涉及用例设计的测试范围**

**并非所有模块都必须在开始时进行验证。**尽管理想情况下需要测试整个模块，但我们只能专注于验证一些核心模块的行为，至少要确定它们的稳定性。通过这样做，它可以保持一定程度的可测试性，以便您可以基于现有的基础设施扩展测试覆盖范围。

*   **根据测试结果细化要求**

嗯，并不是所有的业务需求在被纳入测试任务时都是合理的。让经理们知道测试用例对于防止不合理的需求被发布到市场是至关重要的，这将导致不友好的体验。

# 蔻驰杂音

总而言之，尽管测试工作让工程师感到沮丧，但是仍然有其他方法来吸引经理们的注意力，将测试任务包含到真正的开发周期中。

顺便说一下，任何障碍都不应该成为寻求可行软件和可测试软件的借口。我不喜欢从市场上赚快钱，也不喜欢忽视引导人们建造伟大作品的基本支柱。
