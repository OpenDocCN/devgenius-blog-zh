第二届 Scrum 培训回顾

原文：https://blog.devgenius.io/the-2nd-scrum-training-retrospective-e48bb3270211?source=collection_archive---------10-----------------------#2022-10-21

突然的转变

这个月，我在职业生涯中遇到了一个巨大的转变——我被收购，担任另外两个不同团队的 TPM(技术项目经理),旨在打造顶级加密货币交易所。我不得不离开我以前的 Scrum 团队，把所有的事情留给他们去处理。

令我好奇的是，我有机会一窥交易所在日常运作中通常会做些什么。在正式转到目前的部门之前，我被要求介绍什么是 Scrum，并指导新的合作伙伴在我们目前的场景中采用 Scrum 框架。

正式课程，而不是经验培训过程

嗯，这对我来说是另一个挑战，因为上次我设计了一个实证活动，让成员体验 Scrum 框架下的力量。最初，我尝试头脑风暴其他有趣的活动来引导培训。

尽管如此，我最终还是放弃了，因为无尽的工作让我筋疲力尽。所以我决定用另一种方式来做——做一个关于 Scrum 团队应该是什么样子以及他们应该遵守什么样的原则的演示。

我对任何关于 Scrum 的问题都持开放态度，更不用说一些“怪异的观点”，这可能会引起人们的争议。

与之前的培训不同，这次我删除了一半的幻灯片，帮助观众专注于一些核心概念，而不是吸收太多相关但非核心的想法。

成员和我之间可预见的冲突

我们都有上学的共同经历。当我们上课时，特别是对亚洲学生，我们倾向于在课堂上保持沉默，被动地吸收我们所学的知识。我更愿意接受与成员的一些冲突，而不是在培训期间充当演讲机器。

幸运的是，确实有人试图挑战我对 Scrum 的观点，这给了我另一个机会来反思我是否对它有足够的理解。这些都是你可能会想到的好问题，但在大多数情况下，人们不会公开提出这些问题。

在我看来，在培训课程中有 3 个有趣的问题需要解决。

  • 产品积压为文件

敏捷宣言强调可工作的软件多于文档。一位会员问了我一个问题:这是否意味着我们在尝试制作产品时不必事先写任何文件？

嗯，在 Scrum 中，产品所有者应该有序地维护产品积压，并确保它们的优先级。我们可以将这些积压视为另一种形式的文档，因为它们代表了功能的最低可行需求。我们的目标是迭代地、增量地交付价值，但这并不意味着我们不需要文档。

事实上，对于那些之后要接手产品的人来说，写文档还是很重要的。但是我们必须首先专注于创造业务价值，这样才能获得快速反馈来改进当前流程，减少部分完成的工作停留时间过长。

综上所述，编辑最少的技术文档取决于场景。Scrum 并不认为这是一个必要的步骤，而是将价值交付放在首位。作为一名软件工程师，可读的文档帮助其他人更快地控制产品的愿景，我们不会否认它的必要性。

  • 紧密跟踪与松散跟踪

第二个问题似乎更有争议，因为它直接反映了团队过去对进度跟踪的反应。当我表示我没有对所有成员的进展进行深入跟踪时，有人指责我不知道管理的核心精神，而是花费了他们大量的时间来听我的陈述。

从他的角度来看，他认为需要紧密的进度跟踪来保持高交付率，因为人们倾向于拖延。我们只需要积极地要求人们按时交货，一切都会按预期进行。

当然，在某些情况下，我们不得不强迫人们满足我们的进步，否则目标将无法实现。例如，如果有一个预定的活动将在一周内举行，而相应的系统没有得到验证。我们必须让工程师们想出一些最低限度的解决方案来确保它的正确性。

然而，如果我们总是强迫员工按时完成目标，你能想象会有什么后果吗？从我过去的经验来看，那些习惯于被推动的人没有承担足够的责任来完成他们的任务，因为他们已经习惯于被别人要求。他们不负责他们应该拿什么，这对一个团队的自我管理来说是一场灾难。

我认为，短期的紧追不放会带来即时的效果，但长远来说，会令议员对他们的工作不负责任。一个 scrum 大师应该帮助他们消除障碍，这样可以帮助他们专注于解决他们应该面对的问题。

总之，我不否认密集跟踪对人们的直接影响，但我更喜欢培养人们对他们的任务的责任感。因此，我不会采取无休止的跟踪，除非是在一些紧急情况下需要。

  • 剧情估计

当谈到故事点估算时，我一次又一次地强调，实际上这个行动旨在引发人们对某个问题的讨论，这样他们就可以具体地描绘出它包含了什么样的任务。故事点和花费的时间这两个概念应该分开。

有些人可能会认为需要时间估计，这样他们就可以预测某个问题何时能够解决。对我来说，我只是把时间估计作为一个参考，帮助产品负责人审查交付的隐藏风险，并对它们做出反应。

本质上，故事点的评估帮助我回顾所有问题的消耗，这样我就可以从中发现有问题的过程并进行优化。主要焦点仍然是向最终用户交付有价值的、可行的产品。如果人们理解了它的核心思想，他们就不会被术语的外表所困，而是会专注于探索故事中的任务。

蔻驰杂音

Scrum 暴露问题，但不解决问题

我敢打赌，大多数人可能会在某处读到类似的句子。需要指出的是，任何敏捷框架，甚至是 Scrum，都不会自己解决问题。它只是通过各种活动和原则来暴露它们。最终，人们不得不面对并解决这些问题。

我真的很感激，我再次有机会重新思考框架的一些核心价值观，这也澄清了我对团队建设的内在心态。
