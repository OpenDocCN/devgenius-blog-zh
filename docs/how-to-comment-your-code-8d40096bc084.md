# 要点

1.  **评论是一种负担-** 这是我想让你实现的心态。评论是你和未来开发者的负担。当您更新代码库时，需要对它们进行维护和更新。它们会占用你的文件空间，增加你需要搜索的字符数。注释是你系统的负担，盲目添加注释是个坏主意。然而，你仍然需要它们。要争取的理想是评论越少越好。
2.  **用描述性的变量/函数名替换注释-** 这是我见过很多的事情(尤其是那些听说过注释重要性的过分热心的初级开发人员)。他们为函数和简单的例程添加注释。尽可能用描述性的函数/变量名替换注释。如果你有一个运行计算的登录，用描述性的名字存储结果(最终的和中间的),这样你就可以清楚地知道你的例程在做什么，而不需要添加注释。
3.  **不要评论什么，评论为什么——**我看过的很多评论都花了很大力气解释代码做了什么。几乎没有人解释原因。在您的文档中花些时间解释这段代码做什么，以及它如何嵌入到更大的系统中。这可能在 docstring 和/或您的中央文档系统中(如果您的组织有这样的系统的话)。
4.  **解释输入和输出**——对文档字符串最简单的调整之一就是解释函数的输入和输出。如果您有描述性的名称，这个过程就变得非常简单。这对于希望理解流程的人来说很有帮助。你能做的投资回报率最高的改变之一。
